# CONTENT_ARCHITECTURE_V1

Архитектура контентного слоя `theglavnoe.com`. Описание принципов и связей между типами страниц. Без реальных URL — только архитектура.

Опорные документы (не дублируем, ссылаемся):
- `CASE_TEMPLATE_V1.md` — шаблон кейс-страницы
- `ARTICLES_MAP_REAL_ESTATE.md` — карта блог-статей и шаблон Guide
- `CONTENT_MAP_FULL_SITE.md` — постраничная карта 25 страниц
- `/root/ops/CASE_SOURCES_OF_TRUTH.md` — источники цифр для кейсов

---

## 1. Типы страниц контентного слоя

Сайт делится на **презентационный слой** (главная, о, контакты, портфолио) и **контентный слой** — страницы, которые работают на воронку и SEO. Контентный слой — это то, ради чего пользователь возвращается и через что Google приводит трафик.

| Тип | Где живёт | Сколько | Что делает |
|-----|-----------|---------|------------|
| **Product** | `/product/<slug>/` | 5 | Продаёт услугу. Конвертит горячих клиентов в заявку |
| **Case** | `/work/<slug>/` или `/cases/portfolio/<slug>/` | 3 текущих + 2-3 на вынос из блога | Доказывает что продукт работает. Снимает возражения |
| **Guide** | `/blog/<slug>/` | 3 текущих + 5 запланированных | Прогревает холодный трафик из Google, ведёт в воронку |
| **Framework** | `/blog/<slug>/` (как подтип Guide) | 0 текущих, 1-2 запланированных | Методологические разборы («Что такое квал-лид», «Бюджет маркетинга 2026»). Авторитет + SEO |

**Что НЕ контентный слой** (упоминаем для границы): `/` (главная), `/about/`, `/contact/`, `/cases/` (листинг), `/blog/` (листинг), `/404`.

## 2. Чем отличаются Product / Case / Guide / Framework

### 2.1 Сравнительная таблица

| Параметр | Product | Case | Guide | Framework |
|----------|---------|------|-------|-----------|
| **Цель страницы** | Получить заявку на услугу | Доказать что услуга работает на цифрах | Объяснить методику, прогреть | Дать систему мышления, утвердить экспертизу |
| **Длина / глубина** | 1.5-2 экрана scroll, плотно | 4-6 экранов scroll, средняя глубина | 6-10 экранов scroll, большая глубина | 6-10 экранов, методологическая глубина |
| **Тон** | Продающий, конкретный | Разбор: «было → сделали → стало» | Обучающий, пошаговый | Концептуальный, фреймворковый |
| **Доказательная база** | Тарифы, состав работ, метрика продукта, 1 цитата | Цифры из CRM/рекламных кабинетов, скрины, цитата клиента, СМИ-публикация | Скрины примеров, схемы, сравнительные таблицы, ссылки на 1-2 кейса как proof | Схемы, формулы, бенчмарки, чек-листы, ссылки на кейсы как иллюстрации |
| **ЦА (этап воронки)** | Готов к разговору (горячий) | Сравнивает подрядчиков (тёплый) | Изучает рынок (тёплый-холодный) | Холодный (Google) или эксперт (профи) |
| **SEO-цель** | Transactional («лидген telegram застройщик», «авито реклама агентство») | Mixed (transactional + informational) — «кейс telegram ads недвижимость» | Informational («как застройщику вести telegram», «стратегия маркетинга 2026») | Informational + Navigational («что такое квал-лид», «бюджет маркетинга недвижимости») |
| **CTA-тип** | Hard CTA (заполнить бриф, написать в TG) | Hard + Secondary (тот же продукт + другой кейс) | Soft в середине + Hard в конце | Soft (подписаться на канал + ссылка на кейс) |
| **Источник цифр** | Прайс-карта, состав тарифов | `CASE_SOURCES_OF_TRUTH.md` | Открытые исследования + наш опыт | Открытые исследования + методология агентства |
| **Обновление** | Раз в квартал (цены) | Раз в полгода (новые цифры) | Раз в год | Раз в год |

### 2.2 Типовая структура по разделам

#### Product (по PRODUCT_TEMPLATE_SPEC)
```
Hero (заголовок + 1 ключевая цифра продукта)
→ Описание услуги (1-2 абзаца)
→ 3 тарифа (Tier 1 / Tier 2 / Tier 3) + чипы «что входит»
→ Метрики продукта (3-4 цифры по образцу 3600+/1M+/20K+/№1)
→ Связанный кейс (1 карточка с цифрой и переходом)
→ FAQ (3-5 вопросов)
→ CTA «Стартовать» (Hard)
```

#### Case (по CASE_TEMPLATE_V1)
```
Hero (результат + клиент + контекст)
→ Контекст и вводные (3 карточки: Объект / Точка А / Гипотеза)
→ Воронка сквозной аналитики
→ 4 этапа работы (01 Изучили → 02 Придумали → 03 Запустили → 04 Получили)
→ Метрики (4-6 блоков)
→ Графики динамики
→ Цитата клиента
→ CTA «Хочу такой же результат» (Hard)
→ СМИ-публикация (если есть)
→ Связанные кейсы (2-3 карточки)
```

#### Guide (по ARTICLES_MAP_REAL_ESTATE → GUIDE PAGE TEMPLATE)
```
Sticky nav + progress bar
→ Hero (бейдж «Гайд», H1, что внутри, SVG-путь)
→ TOC sticky слева (desktop) / collapsible (mobile)
→ Intro «Что изменилось» (двухколоночник Раньше vs Теперь)
→ 4-6 нумерованных шагов (01-06):
   - большая цифра слева
   - двухколоночник «Что делать» / «Главные ловушки»
   - try-this box в конце шага
→ Цитата эксперта/клиента (между шагами)
→ Key Takeaway (тёмная секция)
→ About author / About agency
→ CTA на продукт (Hard) + 2-3 связанные карточки
```

#### Framework
Та же база что Guide, но:
- меньше «шагов», больше схем и таблиц
- центральный артефакт — SVG-схема фреймворка или чек-лист
- TOC по компонентам фреймворка, не по этапам
- CTA — soft (на канал, на гайд по применению), Hard опционально

### 2.3 Подсказка для авторов

Если страница отвечает на вопрос **«Купить?»** — это Product.
Если **«У них реально получается?»** — Case.
Если **«Как это делается?»** — Guide.
Если **«Как про это думать?»** — Framework.

## 3. Путь пользователя между типами

### 3.1 Главные входы из внешнего мира

```
                       ┌──────────────┐
                       │  Google SEO  │
                       └──────┬───────┘
                              │
              ┌───────────────┼────────────────┐
              ↓               ↓                ↓
          [GUIDE]         [FRAMEWORK]       [CASE]
       (большинство)      (методичка)    (по запросу с
        холодный            холодный      названием клиента
                                          или сегмента)

                       ┌──────────────┐
                       │ TG канал /   │
                       │ Прямой трафик│
                       └──────┬───────┘
                              │
                       ┌──────┴───────┐
                       ↓              ↓
                   [PRODUCT]        [CASE]
                   (горячий)       (тёплый)
```

### 3.2 Маршрут «холодный → заявка»

```
Google
  ↓
[GUIDE] — читает пошаговое объяснение, доверяет
  ↓ (soft CTA в середине: «Так делали в кейсе X»)
[CASE] — видит цифры и доказательство
  ↓ (Hard CTA: «Хочу такой же результат → продукт Y»)
[PRODUCT] — выбирает тариф
  ↓ (Hard CTA: «Стартовать» → @ksandrbloger)
ЗАЯВКА
```

### 3.3 Маршрут «горячий → заявка» (короткий)

```
Прямая ссылка / реклама / TG канал
  ↓
[PRODUCT] — сразу читает тарифы
  ↓
ЗАЯВКА (или → CASE для финальной проверки → ЗАЯВКА)
```

### 3.4 Маршрут «сомневающийся» (прогрев)

```
[PRODUCT] — читает, не убеждён
  ↓ (Secondary CTA: «Как это работает на практике»)
[CASE] — смотрит цифры
  ↓ (Secondary CTA: «Разобрать методику»)
[GUIDE] — изучает технику
  ↓ (возвращается на PRODUCT через footer-CTA или через 1-2 дня)
[PRODUCT] → ЗАЯВКА
```

### 3.5 Обратные ветки

| От | К | Когда |
|----|---|-------|
| Product | Guide | Скептику нужна методологическая база («хочу понять как») |
| Product | Case | Скептику нужна реальная цифра («покажите что было») |
| Case | другой Case | Клиент в другом сегменте («у нас не премиум, у нас эконом») |
| Case | Guide | Хочет повторить сам, без агентства |
| Guide | другой Guide | Связанная техника («а сквозная аналитика?») |
| Guide | Framework | Хочет систему, не отдельный шаг |
| Framework | Guide | Нужна конкретика как применить |

## 4. CTA на каждом этапе

### 4.1 Принципы

- **Hard CTA = действие с риском для пользователя** (написать в TG, заполнить бриф, оставить email). Ведёт к заявке.
- **Soft CTA = действие без риска** (подписаться на канал, прочитать ещё одну страницу). Прогревает.
- **Один Hard на одну смысловую секцию.** Не два. Не «или подпишись».
- **Soft можно несколько** — но не подряд и не в hero.
- **CTA-текст конкретный**, не «узнать больше». «Хочу такой же результат → Лидген Телеграм».

### 4.2 Матрица CTA по типу страницы

| Тип | Hard CTA | Soft CTA | Secondary CTA | Где размещать |
|-----|----------|----------|---------------|---------------|
| **Product** | «Стартовать» → `@ksandrbloger` (TG) или форма брифа | Подписка на TG-канал агентства | «Посмотреть кейс этого продукта» | Hard в hero + после тарифов + после FAQ; Secondary рядом с метриками продукта; Soft в футере |
| **Case** | «Хочу такой же результат — [продукт]» → продуктовая страница или сразу TG | Подписка на канал | «Читать другой кейс» (карточка); «Разобрать технику в гайде» | Hard в hero + после метрик + после цитаты + финальный блок; Secondary в «Связанных кейсах»; Soft в footer |
| **Guide** | «Заказать [продукт]» → product page | «Подписаться на канал» (Telegram); «Скачать чек-лист»; «Читать следующий гайд» | «Посмотреть кейс по этой технике» | Soft в середине (1-2 шт между шагами); Hard в финальном Key Takeaway/CTA-блоке; Secondary в связанных карточках |
| **Framework** | Опционально: «Заказать [продукт]» | «Подписаться на канал», «Скачать фреймворк PDF», «Читать применение в гайде» | «Кейс с применением» | Преимущественно Soft + 1 Secondary. Hard опционально, только если фреймворк подводит к продукту |

### 4.3 Soft vs Hard — куда какие

```
        ────────────── ВОРОНКА ──────────────→
        холодный              тёплый              горячий
        [GUIDE/FW]           [CASE]             [PRODUCT]
            │                  │                    │
   SOFT в середине ────┐   ┌── HARD в hero      HARD в hero
   HARD в конце ───────┴───┴── HARD после      HARD после тарифов
                              метрик и цитаты   HARD после FAQ
                              SECONDARY на      SECONDARY на кейс
                              другой кейс
```

### 4.4 Конкретные CTA-формулы (можно использовать как есть)

| Контекст | Текст CTA |
|----------|-----------|
| Product Hero | «Обсудить под мой объект → стартовать» |
| Product после тарифов | «Заполнить бриф под тариф [N]» |
| Product Secondary | «Как это работает: кейс [клиент]» |
| Case Hero | «Хочу такой же результат → [продукт]» |
| Case после метрик | «Посчитать бюджет под мой объём» |
| Case Secondary | «Другой кейс по [продукту]» |
| Guide Soft (середина) | «Так мы это сделали в кейсе [клиент]» |
| Guide Hard (финал) | «Заказать под ключ → [продукт]» |
| Framework Soft | «Подписаться на канал, чтобы не пропустить разбор» |

## 5. Карта связей сущностей (после интеграции во Framer)

### 5.1 Кардинальность связей

```
                    ┌──────────┐
                    │ PRODUCT  │ (5 шт)
                    │  (5 шт)  │
                    └────┬─────┘
                         │ 1 : N
                         ↓
                    ┌──────────┐
                    │   CASE   │ (3 шт + рост)
                    │   3+N    │
                    └────┬─────┘
                         │ N : M
                         ↓
                    ┌──────────┐
                    │  GUIDE   │ (3 текущих + 5 запланированных)
                    │   8+N    │
                    └──────────┘
```

### 5.2 Связи и их обязательность

| Связь | Кардинальность | Обязательность | Что значит |
|-------|----------------|----------------|------------|
| **Product → Case** | 1 продукт : N кейсов | Желательно ≥1 кейс на продукт | На странице продукта — карточка минимум 1 кейса с этим продуктом |
| **Case → Product** | 1 кейс : 1 главный продукт + N дополнительных | Обязательно 1 главный | Кейс ВСЕГДА доказывает один основной продукт; дополнительные упоминаются в этапе 02 «Придумали» |
| **Product → Guide** | 1 продукт : N гайдов | Желательно ≥1 гайд на продукт | Гайды объясняют технику продукта, ведут на него |
| **Guide → Product** | 1 гайд : 1-2 продукта | Обязательно 1 | Финальный CTA гайда ведёт на основной продукт |
| **Case → Guide** | 1 кейс : 1-3 гайда | Опционально | Кейс может ссылаться на гайд «как мы это делаем в общем случае» |
| **Guide → Case** | 1 гайд : 1-2 кейса | Обязательно ≥1 | Гайд завершается на кейсе как доказательстве |
| **Case → Case** | 1 кейс : 2-3 «похожих» | Обязательно ≥2 | Блок «Связанные кейсы» внизу |
| **Guide → Guide** | 1 гайд : 1-3 связанных | Желательно | Блок «Связанные гайды» в footer |
| **Framework → Guide** | 1 фреймворк : N гайдов | Обязательно ≥2 | Фреймворк ссылается на гайды как «вот как это применять» |

### 5.3 Что должно быть связано в данных (поля для Framer CMS)

Для каждой страницы — обязательные поля связей:

**Product (5 полей связей):**
- `related_cases[]` — массив slug кейсов
- `related_guides[]` — массив slug гайдов
- `primary_case` — главный кейс-доказательство (1 шт)
- `next_step_cold` — slug гайда для холодного клиента
- `next_step_warm` — slug кейса для тёплого

**Case (4 поля связей):**
- `primary_product` — главный продукт (1 шт)
- `secondary_products[]` — дополнительные продукты, упомянутые в кейсе
- `related_guides[]` — гайды по технике (1-3)
- `related_cases[]` — похожие кейсы (2-3)

**Guide (4 поля связей):**
- `primary_product` — главный продукт (1 шт)
- `secondary_products[]` — дополнительные (0-1)
- `proof_cases[]` — кейсы как доказательство (1-2)
- `related_guides[]` — связанные гайды (1-3)

**Framework (3 поля):**
- `related_products[]` — продукты, на которые ведёт фреймворк
- `application_guides[]` — гайды по применению (≥2)
- `proof_cases[]` — кейсы с применением (≥1)

### 5.4 Правило целостности

При публикации новой страницы все связи проверяются:
- Если у Product нет `related_cases` — это проблема (нет доказательства)
- Если у Case нет `primary_product` — кейс висит в воздухе
- Если у Guide нет `proof_cases` — это эссе, а не гайд

Эта проверка должна быть в чек-листе перед публикацией (см. `CASE_TEMPLATE_V1.md` → ЧЕК-ЛИСТ).

## 6. Резюме: контентная воронка

```
ГЛАВНАЯ (presentation)
    │
    ├──→ [PRODUCT 5 шт] ←─────── Google (transactional) / TG канал
    │       │                            ↑
    │       ├──→ [CASE 3+ шт] ──────────┤
    │       │       │                    │
    │       │       └──→ [GUIDE 8+ шт] ─┤
    │       │              │            │
    │       │              └──→ [FRAMEWORK] (опционально)
    │       │
    │       ↓
    │     ЗАЯВКА → @ksandrbloger / hello@glavnoe.com
    │
    └──→ /about/, /contact/ (presentation, не контент)
```

Каждый тип знает свою роль:
- **Product** конвертит
- **Case** доказывает
- **Guide** прогревает
- **Framework** утверждает экспертизу

Без одного из четырёх — воронка дырявая.
